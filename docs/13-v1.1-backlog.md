# 13 — v1.1 Backlog

What's explicitly deferred from v1, why, and what should ship in v1.1.

## Philosophy

Deferral is not abandonment. Every item here was considered for v1 and consciously moved out. The reasons fall into three categories:

1. **Needs real-world signal to design well.** Building in advance produces wrong defaults.
2. **Substantial engineering scope.** Would push v1 timeline past viability.
3. **Out of v1 strategic scope.** Doesn't serve the v1 audience (OSS, BYO key, mobile-first).

## v1.1 candidate features

### Full safety harness

**What:** Pluggable safety module with input/output classifiers, policy engine, refusal UX.

**Why deferred:**
- Safety thresholds are use-case specific; one default policy will be wrong for many apps.
- Real refusal UX needs observation of real refusal scenarios.
- Anthropic's built-in safety + the destructive-action gate + content reporting cover the floor.

**v1.1 scope:**
- `harness-safety` module implementing `LLMMiddleware`.
- Pluggable classifiers (rule-based defaults; pluggable for ML-based).
- Policy DSL: per-script policies, per-content-category policies.
- Refusal UX: consistent visual treatment when the app declines.
- Reference safety policy for the Undercurrent reference app.

**Effort estimate:** 6 weeks.

### Full evaluation suite

**What:** Large intent corpus, scoring, A/B testing, regression dashboards.

**Why deferred:**
- Intent corpus must be derived from real usage.
- Scoring strategies depend on real failure modes.
- Dashboards without real data are noise.

**v1.1 scope:**
- Intent corpus management (YAML + version control).
- Multi-turn intent tests.
- Semantic scoring (output similarity, not exact match).
- Per-script regression tracking.
- CI dashboards.
- Optional: A/B prompt testing in CI.

**Effort estimate:** 4 weeks.

### Model-agnostic abstraction

**What:** Support for OpenAI, Ollama, OpenRouter, etc. via additional `LLMClient` implementations.

**Why deferred:**
- Premature abstraction. Ship one model well first.
- Cross-provider tool format translation is non-trivial.
- v1 audience is comfortable with Claude.

**v1.1 scope:**
- `llm-openai` module.
- `llm-ollama` module (on-device option).
- Tool format translation layer.
- Capability negotiation (some providers don't support parallel tool use).
- Per-provider model catalog and pricing.

**Effort estimate:** 4 weeks.

### Automatic background memory

**What:** Agent infers what's worth remembering and stores without explicit `memory.store` calls.

**Why deferred:**
- Privacy-sensitive; explicit-only is safer to ship first.
- Wrong defaults could erode trust permanently.
- Need to observe what users actually want remembered before designing the inference.

**v1.1 scope:**
- Inference layer that proposes memories during a turn.
- User confirmation flow ("I'd like to remember that — OK?").
- Automatic-with-batched-confirmation mode for power users.
- Promotion path from session → permanent memory.
- Decay / aging for low-value memories.

**Effort estimate:** 4 weeks.

### Semantic memory recall (embeddings)

**What:** Replace FTS5 lexical search with embedding-based semantic search.

**Why deferred:**
- Adds a dependency (embedding model) that bloats app size.
- Lexical search is good enough for explicit user-curated memories.
- On-device embedding adds latency.

**v1.1 scope:**
- On-device embedding model (small, mobile-optimized).
- Or: cloud embedding via Anthropic if/when offered.
- Hybrid search: lexical + semantic.
- Backwards-compatible with v1 memories.

**Effort estimate:** 3 weeks.

### Streaming agent responses

**What:** Stream LLM responses to the user as they're generated, instead of waiting for full response.

**Why deferred:**
- Tool-use streaming is complicated (partial JSON in tool calls).
- v1 architecture is correct without it.
- UX win is real but not gating.

**v1.1 scope:**
- Streaming `LLMClient` support.
- Streaming-aware middleware (observability records final, others operate on completed response).
- UI integration: progressive text rendering in chat templates.

**Effort estimate:** 3 weeks.

### MCP client integration

**What:** Substrate can consume MCP servers as additional tool sources.

**Why deferred:**
- v1 has its own script catalog; MCP is additive.
- Needs careful security model (MCP servers are external trust boundaries).
- Existing user value is high but not gating.

**v1.1 scope:**
- MCP client module (`llm-mcp` or `scripts-mcp`).
- User configures MCP servers in settings.
- Per-server permission scoping.
- Tool list dynamically extended with MCP server tools.

**Effort estimate:** 4 weeks.

### iOS Shortcuts integration

**What:** Substrate scripts callable from iOS Shortcuts; iOS Shortcuts callable as substrate scripts.

**Why deferred:**
- Platform-specific; doesn't move the cross-platform substrate.
- Power-user feature.

**v1.1 scope:**
- AppIntents declarations for substrate's core scripts.
- A `run_shortcut` substrate script invokable by the agent.
- Documentation for users on integrating substrate apps with Shortcuts.

**Effort estimate:** 2 weeks.

### Android Tasker integration

**What:** Symmetric to iOS Shortcuts — Tasker as orchestration backend.

**Why deferred:** Same reasoning as Shortcuts.

**v1.1 scope:**
- AutoTools / Tasker plugin.
- HTTP-callable substrate scripts (so Tasker can invoke them).
- A `run_tasker_task` substrate script.

**Effort estimate:** 1 week.

### QR-from-desktop key transfer

**What:** Avoid typing API keys on mobile by transferring from desktop via QR.

**Why deferred:** Nice UX improvement, not gating. Paste flow ships in v1.

**v1.1 scope:**
- Desktop companion (web-based) that generates QR.
- Mobile app scans QR; key flows through camera into Keystore/Keychain.
- Web companion has no servers; key flows peer-to-peer via QR.

**Effort estimate:** 2 weeks.

### On-device LLM models

**What:** Substrate can run with a local model (Llama, Gemma, Phi) instead of Claude.

**Why deferred:**
- Quality gap is large for the kind of agentic reasoning the substrate expects.
- App size impact significant.
- Different latency characteristics complicate testing.

**v1.1 scope:**
- `llm-local` module wrapping llama.cpp or similar.
- Model selection in settings (download from Hugging Face).
- Reduced tool catalog / simpler prompts for smaller models.
- Hybrid mode: local for simple, remote for complex.

**Effort estimate:** 6+ weeks.

### Voice I/O

**What:** Speak to the app; hear responses.

**Why deferred:**
- Adds a substantial dependency surface (STT/TTS).
- Different UX model deserves its own treatment.

**v1.1 scope:**
- On-device STT (iOS Speech / Android SpeechRecognizer).
- On-device TTS (system).
- Optional: cloud STT/TTS for better quality.
- Voice UI mode in chat templates.

**Effort estimate:** 3 weeks.

### Multimodal inputs (images)

**What:** Send images to Claude as part of conversations; LLM can reason about them.

**Why deferred:**
- v1 ships text-only flows; multimodal is a separate UX track.
- Increases token costs significantly.

**v1.1 scope:**
- Image attachment in chat templates.
- Image content blocks in `LLMRequest`.
- Cost middleware updated to account for image token costs.
- Image storage and retention policy.

**Effort estimate:** 2 weeks.

### Reproducible builds (verified end-to-end)

**What:** Verify the published binary matches the source byte-for-byte.

**Why deferred (verification, not implementation):**
- Documentation for the path ships in v1.
- Actual verification requires F-Droid pipeline + iOS workarounds.
- Important for long-term trust but not gating.

**v1.1 scope:**
- F-Droid build successfully reproduces release binaries.
- Document iOS reproducibility approach (likely partial, given signing).
- Build attestation via sigstore.

**Effort estimate:** 2 weeks.

### Third-party app showcase

**What:** A directory of apps built on the substrate.

**Why deferred:** Needs third-party apps to exist first.

**v1.1 scope:**
- Submission process for showcase entries.
- Documentation site directory.
- Quality criteria.

**Effort estimate:** 1 week (mostly ongoing curation).

### Plugin marketplace

**What:** User-installable script/template plugins.

**Why deferred (probably v2.0, not v1.1):**
- Significant security model required.
- Substrate maturity needed first.
- Distribution and discoverability are big problems.

**v1.1 might:** define the plugin manifest format and a side-loading mechanism for power users, no marketplace.

## v1.1 timeline sketch

If v1 ships at month 7.5, v1.1 could ship at month 11–12. Priority order:

| Order | Feature | Effort |
|---|---|---|
| 1 | Full safety harness | 6 wk |
| 2 | MCP client integration | 4 wk |
| 3 | Streaming | 3 wk |
| 4 | Full evaluation suite | 4 wk |
| 5 | Automatic memory (with confirmation flow) | 4 wk |

That's 21 weeks of work. With slip and review, a realistic v1.1 ships in ~5–6 months after v1.

Other items push to v1.2 or beyond based on community signal.

## How v1.1 priorities will be set

1. Real usage data from v1 (telemetry opt-in, user feedback, GitHub issues).
2. Contributor interest (someone willing to own a module gets weight).
3. Strategic value (does it enable new app categories? does it unblock commercial path?).
4. Cost of deferring further (some things compound — safety holes left longer get worse).

Open governance: published list, public discussion, tech lead final call.

## What stays "never" (or much later)

Some things probably never belong in the substrate, even at v2.0:

- **Generic IDE / app builder.** Different product.
- **Server-rendered UI / hot reload.** Native is the value prop.
- **Multi-user / collaboration.** Personal-agent shape doesn't generalize.
- **Built-in payment processing.** Apps that want this build it themselves.
- **Cloud sync as a substrate feature.** Apps handle their own data sync if they want it.
- **Generic LLM proxy / API gateway.** That's a different product (could be a commercial offering, but not substrate).

Naming these out keeps scope honest.

## What might be the basis of a commercial v2

The substrate's commercial commercial path, if you decide to take it, layers on top of v1 (and post-v1 OSS work):

- Managed `LLMClient` with handled billing.
- Hosted shared MCP servers.
- Team / multi-user features.
- Enterprise SSO + admin console.
- Compliance certifications (SOC 2, HIPAA paths).
- White-label deployment for app developers.

These are not v1 or v1.1. They're a separate product decision. Flagged here for completeness; revisit after v1 is shipping and you have real-user signal.
